# SampleOrderSystem

가상의 반도체 회사 "S-Semi"의 시료(Sample) 생산 주문을 등록 → 주문 → 승인/거절 → 생산 → 출고까지
관리하는 콘솔 기반(메뉴 + 표준입력) 애플리케이션입니다.

## 실행 방법

```
python main.py
```

메인 메뉴가 콘솔에 표시되며, 번호를 입력해 기능을 선택합니다. Windows 콘솔(코드페이지 949 등)에서
실행하더라도 한글 입출력이 깨지지 않도록 시작 시 표준입출력을 UTF-8로 강제 재구성합니다(`main.py`의
`_force_utf8_io()`) — 별도 플래그를 줄 필요는 없습니다.

## 테스트 실행 방법

```
pip install -r requirements-dev.txt
pytest
```

## 더미 데이터 생성

메뉴를 매번 손으로 입력하지 않고 대량의 초기 시료/주문 데이터를 채우려면 다음 커맨드를 사용합니다:

```
python generate_dummy_data.py --samples 20 --orders 50 --seed 42
```

- `--samples`: 생성할 시료 개수
- `--orders`: 생성할 주문 개수 (항상 `RESERVED` 상태로 생성됩니다)
- `--seed`: 재현 가능한 결과를 위한 랜덤 시드 (생략 가능)

생성된 데이터는 `python main.py`가 사용하는 것과 동일한 `data/samples.json`, `data/orders.json`에
저장되므로, 생성 직후 바로 콘솔 메뉴(시료 관리, 모니터링 등)에서 확인할 수 있습니다. 단, 이 도구는
`RESERVED` 주문까지만 생성하며 `CONFIRMED`/`PRODUCING`/`RELEASE`/`REJECTED` 상태는 실제 메뉴(주문
승인/거절, 생산 라인, 출고 처리)를 통해서만 만들어집니다.

## 디렉터리 구조

```
app/
  model/        # Sample/Order 엔티티, 생산 큐 등 도메인 모델
  persistence/  # JSON 파일 기반 영속성 및 CRUD 리포지토리
  controller/    # 주문 생명주기 로직 (승인 분기, 생산 스케줄링, 출고 처리 등)
  view/          # 콘솔 메뉴 루프와 화면 렌더링
  generator/     # 더미 데이터 생성기 (Model에 반영)
main.py                    # 콘솔 애플리케이션 진입점
generate_dummy_data.py     # 더미 데이터 생성 CLI 진입점
data/                       # samples.json, orders.json (실행 시 자동 생성)
tests/                      # pytest 테스트 스위트
```

## 명세(PDF) 대비 구현 일치성 검토

`pdf/[CRA_AI] Day3_개인과제_반도체시료관리_r1 2.pdf` 원본 명세와 현재 구현을 대조 검토한 결과입니다.

**생산 완료 처리 방식(수동 트리거) — 그대로 유지**: 명세는 생산 완료 시 `PRODUCING → CONFIRMED`
전환만 요구할 뿐, 완료를 "언제/어떻게" 트리거할지는 정하지 않았습니다(`docs/PLAN.md` Phase 4도 "자동
타이머든, 수동 '다음 단계 진행' 커맨드든 방식은 구현 시 결정"이라고 명시). 생산라인 화면 예시(PDF
21쪽)에 실시간 진행률(%)·완료 예정 시각이 보이지만, 해당 쪽 각주가 "화면 구성은 자유롭게 결정"이라고
분명히 밝히고 있어 강제 요구사항이 아닙니다. 자동 스케줄러(백그라운드 시간 경과에 따른 자동 완료)
도입을 검토했으나, 명세를 충족하기 위해 반드시 필요한 것은 아니고 이미 구현·테스트된 수동 트리거
방식(생산 라인 조회 화면에서 "완료 처리" 선택)이 명세를 온전히 만족하므로 그대로 유지하기로 했습니다.

**의도적으로 남겨둔 두 가지 차이점**:

1. **시료 등록 시 "시료 ID" 직접 입력** — PDF 12쪽은 시료 등록 입력값에 "시료 ID"를 포함하지만,
   이 프로젝트는 Phase 0/1에서 사용자가 직접 입력할 경우 발생할 수 있는 ID 충돌 문제를 원천 차단하기
   위해 시스템이 자동 채번(`S-001`, `S-002`, ...)하도록 결정했습니다(`docs/design/phase0.md`,
   `docs/design/phase1.md`, `docs/FEATURES/01-sample.md`에 근거 기록).
2. **주문 승인 화면의 확인 순서** — PDF 17쪽 예시는 승인 선택 후 부족분/실생산량/총생산시간을 먼저
   보여준 다음 `[Y]/[N]`을 다시 묻는 2단계 흐름이지만, 이 프로젝트는 `[Y]/[N]`을 먼저 묻고 계산
   결과는 처리 완료 메시지에만 표시합니다. 두 방식 모두 해당 쪽 각주("화면 구성은 자유")상 허용되는
   범위이며, 필요 시 나중에 조정 가능한 UX 선택 사항입니다.

그 외 메뉴 구성, 상태 전이(`RESERVED`/`REJECTED`/`PRODUCING`/`CONFIRMED`/`RELEASE`), 승인 분기 로직,
`ceil(부족분/수율)` 산식, FIFO 생산 큐, 모니터링의 REJECTED 제외·여유/부족/고갈 판정, 출고 처리 흐름은
PDF 명세와 정확히 일치합니다.

## 채점 기준 자체 평가

미션 명세(PDF Chapter 3)가 밝힌 평가 주안점 5가지에 대해, 실제로 무엇을 했고 어떤 한계가 있는지
스스로 평가한 내용입니다. 좋은 점만 나열하지 않고 부족한 부분도 함께 적었습니다.

### 1. 문서 관리 (CLAUDE.md, PRD.md 등)

- `CLAUDE.md`(아키텍처 가이드·도메인 규칙) → `docs/PRD.md`(요구사항 요약) → `docs/FEATURES/01~06`(기능별
  상세 명세) → `docs/design/phase0~7.md`(Phase별 구현 설계, 8개 문서) 순으로 계층을 나눠 관리했고,
  각 문서가 서로 반복 서술하지 않고 상위/하위 문서를 참조하도록 구성했습니다.
- Phase를 구현하기 전마다 설계 문서를 먼저 작성하고, `doc-consistency-checker` 서브에이전트로 문서 간
  (그리고 문서-코드 간) 정합성을 점검한 뒤에야 구현에 들어갔습니다. 이 과정에서 실제로 여러 번 불일치를
  찾아 바로잡았습니다 — 예: Phase 1 설계 당시 "시료 ID 자동 채번" 결정이 `docs/PLAN.md`/
  `docs/FEATURES/01-sample.md`의 "사용자가 ID를 입력한다"는 서술과 어긋난 것을 발견해 두 문서를 함께
  수정, Phase 7 설계 시 `CLAUDE.md`의 "프로젝트 현황"이 Phase 0 시점에 멈춰 있던 것을 발견해 최신화.
- **한계**: 문서 정합성 점검이 대부분 "다음 Phase 설계 시점"에 트리거되는 구조라, Phase 사이사이에
  발생한 사소한 표현 오차(예: 설계 문서에 적힌 누적 테스트 개수가 실제와 몇 개 어긋난 경우)는 스스로
  중요하지 않다고 판단해 넘어간 것들이 있습니다 — 채점자가 문서를 정독하면 발견할 수 있는 수준의
  자잘한 오차입니다.

### 2. Harness 도입

- `.claude/agents/`에 역할이 분리된 서브에이전트 3종을 정의해 사용했습니다: `doc-consistency-checker`
  (읽기 전용, 문서/코드 정합성만 검증), `action-implementer`(설계 문서를 그대로 구현, 테스트 작성 안 함),
  `verify-tester`(pytest 작성/실행 + 콘솔 실제 구동 검증, 버그를 직접 고치지 않고 보고만 함).
- Phase 0~7 전체에 걸쳐 "문서 확인 → 구현 → 검증"이라는 동일한 순서를 일관되게 적용했습니다. 특히
  `verify-tester`가 코드만 읽고 판단하지 않고 실제로 `main.py`를 파이프 입력으로 구동해 검증하도록
  강제한 덕분에, 코드 리뷰만으로는 못 잡았을 실제 버그(Windows cp949 콘솔에서 저장 실패가 "중복 ID
  오류"로 오인 표시되던 문제, Phase 1)를 실행 검증 단계에서 잡아낼 수 있었습니다.
- **한계**: 세 에이전트 모두 이 세션(저) 안에서 순차적으로 호출한 것이지, 완전히 독립된 검증 주체는
  아닙니다 — 즉 "같은 모델이 역할을 바꿔가며 스스로 점검"하는 구조의 한계(같은 맹점을 셋 다 놓칠
  가능성)는 여전히 남아 있습니다.

### 3. Test

- pytest 149개(Phase별 단위 테스트 + `test_end_to_end_flow.py` 통합 테스트 1건)가 있으며 전부 통과
  합니다. `unittest.TestCase` 대신 pytest 스타일(함수 + `assert` + `pytest.fixture`/`tmp_path` +
  `pytest.raises`)로 일관되게 작성했습니다.
- 이 프로젝트의 핵심 로직(주문 승인 재고 분기, 수율 기반 생산량 계산의 부동소수점 경계값, FIFO 순서,
  REJECTED 모니터링 제외)은 콘솔 입출력과 분리된 순수 로직으로 설계해, stdin 시뮬레이션 없이 테스트
  가능하도록 만들었습니다(`CLAUDE.md`가 요구한 방향).
- Phase 5에서 사람이 손으로만 확인했던 "주문 생성→승인→생산→출고" 전체 흐름을 Phase 7에서 자동화된
  통합 테스트로 옮겼습니다.
- **한계**: 테스트가 대부분 Controller 단위로 구성되어 있고, `main.py`/`MainController`의 메뉴 배선
  자체를 검증하는 자동화된 테스트는 없습니다(사람이 직접 콘솔을 구동해서만 확인). 또한 커버리지
  측정 도구(`coverage.py` 등)를 붙이지 않아 "몇 %가 테스트되는지"는 정량적으로 보여드릴 수 없습니다.

### 4. CleanCode

- Model(`app/model`)/Persistence(`app/persistence`)/Controller(`app/controller`)/View(`app/view`)
  계층을 분리했고, 어느 Controller도 `input()`/`print()`를 직접 호출하지 않고 항상 View를 통해서만
  콘솔과 상호작용합니다 — View에는 재고 계산이나 상태 전이 같은 비즈니스 로직이 없습니다.
- 주석은 "왜"가 비직관적인 경우(부동소수점 보정 이유, ConsoleMVC 원본과 다르게 구현한 이유 등)에만
  달았고, 코드가 스스로 설명하는 내용은 반복 서술하지 않았습니다.
- **한계**: 이 저장소에는 타입 힌트가 일부(주로 생성자 인자)에만 적용되어 있고 전체 함수에 일관되게
  적용되어 있지 않습니다. 또한 `ruff`/`black`/`mypy` 같은 린터·포매터·타입체커 설정 파일이 하나도
  없어서, 코드 스타일 일관성이 "직접 리뷰로 유지"되는 수준이지 도구로 강제되는 수준은 아닙니다. 이
  부분은 실제로 개선 여지가 있는 지점이라고 판단합니다.

### 5. Commit 이력

- 총 29개 커밋으로, Phase마다 "설계 문서 커밋 → 구현 커밋"을 분리해서 남겼습니다(예: `ceaad88`
  Phase 3 설계 문서, `9c7ff38` Phase 3 구현). 버그 수정(`81e41e3` 재고 정책 수정)이나 문서 정합성
  보정도 그 자체로 의미가 드러나는 별도 커밋으로 분리했습니다.
- 모든 커밋 메시지는 "무엇을 바꿨는지"보다 "왜 그렇게 바꿨는지"를 설명하는 방향으로 작성했습니다(예:
  `production_controller.py`가 왜 `shortage_qty`가 아니라 `actual_qty` 전체를 반영하는지).
- **한계**: 각 Phase 내부(구현 → 버그 수정 → 재검증)의 세부 커밋 단위가 항상 잘게 쪼개져 있는 것은
  아니라서, 한 커밋 안에 "구현 + 그 구현을 검증하며 나온 개선"이 함께 들어간 경우가 있습니다(더
  잘게 나눌 수도 있었을 부분).
