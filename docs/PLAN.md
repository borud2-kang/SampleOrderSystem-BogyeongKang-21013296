# 개발 Phase 계획 (초안)

각 Phase는 **그 자체로 실행 가능한 콘솔 애플리케이션**이 되는 것을 목표로 한다. Phase가 끝날 때마다
`python main.py`(또는 확정된 실행 커맨드)로 직접 실행해서, 아래 "확인 포인트"에 적힌 시나리오를 사람이 직접
콘솔에 입력해보며 확인할 수 있어야 한다. 다음 Phase로 넘어가기 전에 이 확인이 끝나는 것을 전제로 한다.

각 Phase의 상세 기능 규칙은 `docs/FEATURES/`의 해당 문서를 따른다. 이 문서는 "무엇을, 어떤 순서로 만들고
어떻게 확인할지"에 집중하고, 세부 입력값/엣지 케이스는 반복하지 않는다.

## Phase 0 — 뼈대: MVC 스켈레톤 + 영속성 + 메인 메뉴

**목표**: 화면은 비어 있어도 되지만, 아키텍처와 데이터 저장/로딩 파이프라인이 동작해야 한다.

- Model / Controller / View 패키지 구조 확립 (`../ConsoleMVC` 패턴 참고).
- Sample/Order 데이터 모델과 JSON(또는 선택한 방식) 기반 영속성 계층 연동 (`../DataPersistence` 패턴 참고).
- 메인 메뉴 화면(6개 메뉴 항목 표시)만 존재하고, 각 메뉴는 "준비 중" 정도로 응답해도 무방.
- 프로그램 종료 후 재실행 시 저장된 데이터가 그대로 로드되는지가 핵심.

**확인 포인트 (사람이 볼 부분)**

- [ ] `python main.py` 실행 시 메인 메뉴가 뜬다.
- [ ] 메뉴에서 잘못된 번호를 입력해도 프로그램이 죽지 않고 재입력을 받는다.
- [ ] 프로그램을 종료(`0`)했다가 다시 실행해도 에러 없이 뜬다 (이 시점엔 저장된 데이터가 없어도 정상).
- [ ] `data/` 하위에 저장 파일(예: `samples.json`, `orders.json`)이 생성/유지되는 것을 파일 탐색기로 확인.

## Phase 1 — 시료 관리

**목표**: 시료 등록·조회·검색이 실제로 동작하고, 재시작해도 유지된다.

관련 문서: [`docs/FEATURES/01-sample.md`](FEATURES/01-sample.md)

- 시료 등록 (시료 ID/이름/평균 생산시간/수율 입력, 중복 ID 검증).
- 시료 목록 조회 (현재 재고 함께 표시 — 이 시점 재고는 0 또는 초기값).
- 이름으로 시료 검색.

**확인 포인트**

- [ ] 시료를 2~3개 등록하고, 목록 조회에서 모두 보이는지 확인.
- [ ] 이미 존재하는 시료 ID로 등록 시도 시 거부되는지 확인.
- [ ] 수율에 `0`, 문자열 등 잘못된 값을 입력했을 때 적절히 재입력을 요구하는지 확인.
- [ ] 이름으로 검색해 원하는 시료만 나오는지 확인.
- [ ] 프로그램을 재시작해도 등록한 시료가 그대로 남아있는지 확인.

## Phase 2 — 시료 주문 (예약)

**목표**: 고객 주문을 접수해 `RESERVED` 상태로 저장할 수 있다.

관련 문서: [`docs/FEATURES/02-order.md`](FEATURES/02-order.md)

- 시료 ID/고객명/수량을 입력받아 주문 생성.
- 미등록 시료 ID, 잘못된 수량에 대한 검증.
- 생성된 주문은 항상 `RESERVED` 상태.

**확인 포인트**

- [ ] Phase 1에서 등록한 시료로 주문을 생성하면 주문번호가 발급되고 상태가 `RESERVED`인지 확인.
- [ ] 등록되지 않은 시료 ID로 주문을 시도하면 거부되는지 확인.
- [ ] 수량에 0, 음수, 문자열을 넣었을 때 재입력을 요구하는지 확인.
- [ ] (아직 조회 메뉴가 없다면) 재시작 후 주문 데이터가 유지되는지 데이터 파일로 확인.

## Phase 3 — 주문 승인/거절 (핵심 분기 로직)

**목표**: 이 시스템에서 가장 중요한 로직 — 승인 시 재고에 따라 `CONFIRMED`/`PRODUCING`으로 자동 분기되고,
거절 시 `REJECTED`로 전환된다. (이 시점엔 아직 생산 라인이 실제로 "생산"을 진행하지 않아도 되며, 주문이
`PRODUCING` 큐에 등록되는 것까지만 확인해도 된다 — 실제 생산 진행/완료는 Phase 4.)

관련 문서: [`docs/FEATURES/03-approval.md`](FEATURES/03-approval.md)

- `RESERVED` 주문 목록 표시.
- 승인: 재고 충분 → `CONFIRMED`, 재고 부족 → `PRODUCING`(생산 큐 등록).
- 거절: `REJECTED`로 전환.

**확인 포인트 (가장 꼼꼼히 볼 부분)**

- [ ] 재고를 충분히 넣어둔 시료로 주문 → 승인 시 즉시 `CONFIRMED`로 바뀌는지 확인.
- [ ] 재고가 부족한(또는 재고 0인) 시료로 주문 → 승인 시 `PRODUCING`으로 바뀌는지 확인.
- [ ] 재고 == 주문 수량으로 정확히 같은 경계값에서 "충분"(`CONFIRMED`)으로 처리되는지 확인.
- [ ] 주문을 거절하면 `REJECTED`로 바뀌고, 이미 승인/거절된 주문을 다시 승인/거절하려 하면 막히는지 확인.

## Phase 4 — 생산 라인 (FIFO + 수율 계산)

**목표**: `PRODUCING` 상태 주문이 실제로 생산 큐에서 처리되어 `CONFIRMED`로 전환된다.

관련 문서: [`docs/FEATURES/05-production-line.md`](FEATURES/05-production-line.md)

- 생산 큐(FIFO) 조회: 대기 중인 주문을 순서대로 표시.
- 생산 중 항목의 진행 상황 표시.
- 부족분/수율 기반 실 생산량, 총 생산 시간 계산.
- 생산 완료 처리(자동 타이머든, 수동 "다음 단계 진행" 커맨드든 방식은 구현 시 결정) 시 `PRODUCING →
  CONFIRMED` 전환 및 재고 반영.

**확인 포인트**

- [ ] Phase 3에서 `PRODUCING`으로 전환된 주문이 생산 큐 조회 화면에 FIFO 순서(먼저 승인된 순서)대로
      보이는지 확인.
- [ ] 표시되는 실 생산량이 `ceil(부족분 / 수율)` 계산과 일치하는지 직접 계산해서 검산.
- [ ] 여러 건을 연달아 `PRODUCING`으로 만들었을 때, 먼저 등록된 주문부터 순서대로 처리되는지 확인.
- [ ] 생산 완료 처리 후 해당 주문 상태가 `CONFIRMED`로 바뀌고, 시료 재고가 갱신되는지 확인.

## Phase 5 — 출고 처리

**목표**: `CONFIRMED` 주문을 출고해 `RELEASE`로 마무리한다.

관련 문서: [`docs/FEATURES/06-shipment.md`](FEATURES/06-shipment.md)

- `CONFIRMED` 주문 목록 표시.
- 특정 주문 출고 → `RELEASE` 전환.

**확인 포인트**

- [ ] Phase 3/4에서 `CONFIRMED`가 된 주문들이 출고 목록에 모두 나오는지 확인.
- [ ] 출고 처리 후 해당 주문 상태가 `RELEASE`로 바뀌는지 확인.
- [ ] `CONFIRMED`가 아닌 주문(RESERVED/PRODUCING/REJECTED/이미 RELEASE)은 출고 목록에 나오지 않는지 확인.

이 시점부터 하나의 주문이 `RESERVED → (승인) → CONFIRMED/PRODUCING → CONFIRMED → RELEASE` 전체 흐름을
처음부터 끝까지 콘솔에서 수행할 수 있어야 한다. **전체 플로우를 한 번 손으로 끝까지 실행해보는 것을 이
Phase의 최종 확인으로 삼는다.**

## Phase 6 — 모니터링

**목표**: 지금까지 쌓인 주문/재고 데이터를 요약해서 보여주는 조회 전용 화면을 완성한다.

관련 문서: [`docs/FEATURES/04-monitoring.md`](FEATURES/04-monitoring.md)

- 상태별 주문 건수 (REJECTED 제외).
- 시료별 재고 현황 + 파생 상태(여유/부족/고갈).

**확인 포인트**

- [ ] Phase 1~5에서 만든 주문들의 상태별 건수를 직접 손으로 세어보고 모니터링 화면 숫자와 비교.
- [ ] `REJECTED`로 만든 주문이 있다면, 이 화면의 어떤 집계에도 포함되지 않는지 확인.
- [ ] 재고를 고의로 0으로 만든 시료가 "고갈"로 표시되는지, 넉넉한 시료는 "여유"로 표시되는지 확인.

## Phase 7 — 마무리: Dummy 데이터, 테스트, 문서/커밋 정리

**목표**: 미션 평가 기준(Harness, Test, CleanCode, Commit 이력)을 충족하는 상태로 다듬는다. 기능 추가보다
품질/재현성에 집중하는 단계.

- Dummy 데이터 생성 기능 통합 (`../DummyDataGenerator` 패턴 참고) — 매번 손으로 시나리오를 만들지 않아도
  대량의 시료/주문 샘플 데이터를 채울 수 있게 한다.
- Phase 1~6에서 손으로 확인한 시나리오들을 자동화된 단위/통합 테스트로 옮긴다 (특히 Phase 3, 4의 분기·계산
  로직).
- README/CLAUDE.md/PRD.md가 실제 구현과 어긋나는 부분이 없는지 점검.
- 커밋 이력이 Phase 단위로 의미 있게 나뉘어 있는지 정리.

**확인 포인트**

- [ ] Dummy 데이터 생성 커맨드 실행 후, 시료 관리/모니터링 화면에서 대량 데이터가 정상적으로 보이는지 확인.
- [ ] 테스트 커맨드(예: `pytest`)가 전부 통과하는지 확인.
- [ ] 처음 보는 사람이 README만 보고 실행할 수 있는지(진입점, 실행 커맨드가 명확한지) 확인.

## 진행 방식

Phase는 위 순서(0→7)를 기본으로 하되, Phase 3(승인/거절)과 Phase 4(생산 라인)는 서로 강하게 얽혀 있으므로
Phase 3에서 "PRODUCING 전환까지"만 먼저 확인하고 Phase 4에서 "생산 완료까지" 이어서 확인하는 흐름을
권장한다. 각 Phase 완료 후 커밋을 남겨 Phase별 변경 이력이 명확히 드러나게 한다.
