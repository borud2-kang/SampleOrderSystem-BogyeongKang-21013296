# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 배경 및 문제 정의

가상의 반도체 회사 "S-Semi"는 다양한 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실
등에 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.

- "이 주문 처리됐나요?" — 주문 처리 여부를 확인하기 어려움
- "공정 예약을 했는데 언제 완성되는지 모르겠어요." — 생산 완료 시점 예측 불가
- "이미 충분한 재고가 있는데 왜 추가 공정이 돌아가고 있나요?" — 재고와 생산 현황이 연동되지 않음

엑셀·메모장 기반 수기 관리로 인해 실수가 잦고, 재고·공정 현황을 한눈에 파악할 수 없다. 이를 해결하기 위해
체계적인 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 2. 목표

- 시료 등록부터 주문 접수, 승인/거절, 생산, 출고까지 전 과정을 하나의 시스템에서 추적 가능하게 한다.
- 재고 상태와 생산 현황을 실시간으로 조회할 수 있게 하여, 담당자가 수기 관리 없이 의사결정할 수 있도록 한다.
- 재고 부족 시에만 필요한 만큼 자동으로 생산을 트리거하여, 불필요한 과잉 생산을 방지한다.

## 3. 범위

- 콘솔(CLI) 기반 애플리케이션. GUI/웹 화면은 범위 밖.
- 담당자가 직접 명령을 입력해 시료를 등록하고 주문을 처리하는 단일 프로세스 도구. 별도의 사용자 인증/권한 체계는
  범위 밖 (역할은 개념적 구분일 뿐, 시스템은 하나의 콘솔 세션으로 동작).
- 데이터 영속성(저장 방식은 파일/JSON/DB 등 구현자가 선택)과 CRUD.
- 미션1 PoC(`../ConsoleMVC`, `../DataPersistence`, `../DataMonitor`, `../DummyDataGenerator`, 자세한 내용은
  `CLAUDE.md` 참고)에서 검증된 패턴을 통합하여 완성한다.

### 범위 밖 (Out of scope)

- 웹/GUI 화면, 다중 사용자 동시 접속, 실제 설비 연동(PLC 등), 결제/과금.
- 여러 생산 라인의 병렬 처리 (기본은 단일 라인. 확장이 필요하면 별도 논의 필요).

## 4. 이해관계자 및 역할별 흐름

| 역할 | 책임 |
| --- | --- |
| 고객 | 필요한 시료를 이메일 등으로 요청 (시스템 외부 프로세스) |
| 주문 담당자 | 고객 요청에 맞게 주문서(주문) 작성 → 생산 담당자에게 전달 |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산 관리, 출고 처리 |

```
고객 --(시료 요청)--> 주문 담당자 --(주문서 전달)--> 생산 담당자
고객 <----------------(승인/거절)------------------- 생산 담당자
```

## 5. 핵심 도메인 모델

### 5.1 시료 (Sample)

시스템에 등록된 시료만 주문 가능한, 이 시스템의 가장 기본이 되는 단위.

| 속성 | 설명 |
| --- | --- |
| 시료 ID | 고유 식별자 |
| 이름 | 시료명 |
| 평균 생산시간 | 1개 생산에 소요되는 평균 시간 |
| 수율 | 정상적인 시료 수 / 총 생산 시료 수 (예: 100개 생산 중 90개 정상 = 0.9) |
| 현재 재고 | 즉시 출고 가능한 수량 |

### 5.2 주문 (Order)

| 속성 | 설명 |
| --- | --- |
| 주문번호 | 고유 식별자 |
| 시료 ID | 주문 대상 시료 |
| 고객명 | 주문한 고객 |
| 주문 수량 | 요청 수량 |
| 상태 | 아래 5.3 참고 |

### 5.3 주문 상태 (Order Status)

| 상태 | 의미 |
| --- | --- |
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외 상태, **모니터링에서 제외**) |
| PRODUCING | 주문 승인 완료, 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료, 출고 대기 중 |
| RELEASE | 출고 완료 |

### 5.4 상태 전이도

```
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고 처리)--> RELEASE
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고 처리)--> RELEASE
RESERVED --(거절)--> REJECTED
```

### 6.1 메인 메뉴

- 기능별 선택 화면 표시.
- 전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 등) 표시.

### 6.2 시료 관리

| 기능 | 설명 |
| --- | --- |
| 시료 등록 | 새 시료를 시스템에 추가. 입력값: 시료 ID, 이름, 평균 생산시간, 수율 |
| 시료 조회 | 등록된 모든 시료 목록과 현재 재고 수량 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료 검색 |

### 6.3 시료 주문 (예약)

- 고객이 시료를 요청하면 주문 담당자가 주문을 생성.
- 입력값: 시료 ID, 고객명, 주문 수량.
- 생성 직후 주문 상태는 `RESERVED`.

### 6.4 주문 승인/거절

- **접수된 주문 목록**: `RESERVED` 상태 주문 목록 표시.
- **주문 승인**: 재고 상황에 따라 자동 분기.
  - 재고 충분 → 즉시 `CONFIRMED`.
  - 재고 부족 → 생산 라인에 자동 등록, `PRODUCING`으로 전환.
- **주문 거절**: 즉시 `REJECTED`로 전환.

### 6.5 모니터링

- **주문량 확인**: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 수 표시. `REJECTED`는 유효한
  주문이 아니므로 집계에서 제외.
- **재고량 확인**: 시료별 현재 재고 수량과, 주문 대비 재고 수량에 따른 파생 상태 표시.
  - 여유: 주문 대비 재고 충분
  - 부족: 주문 대비 재고 수량 부족
  - 고갈: 재고 수량이 0

### 6.6 생산 라인

- 생산 라인 정보 표시. 하나의 생산 라인은 시료를 하나씩 생산하며, 주문이 들어온 시료에 대해서만 생산한다.
- **생산량 계산**:
  - 부족분 = 주문 수량 − 현재 재고
  - 실 생산량 = `ceil(부족분 / 수율)` (수율 및 불량률 고려)
  - 총 생산 시간 = 평균 생산시간 × 실 생산량
- 생산 완료 시 주문 상태를 `PRODUCING → CONFIRMED`로 자동 전환.
- **생산 현황 표기**: 현재 생산 중인 시료에 대한 정보(주문 정보, 진행률 등) 표시. 표기 정보 수준은 구현자
  재량.
- **대기 주문 확인**: 생산 큐(대기열)에 있는 주문 목록 표시. 스케줄링 전략은 **FIFO** (선입선출), 우선순위
  재정렬 없음.

### 6.7 출고 처리

- 재고가 충분해진 `CONFIRMED` 주문에 대해 출고 실행.
- 특정 주문을 선택해 출고 처리 → 주문 상태가 `RELEASE`로 전환.

## 7. 비기능 요구사항

- **데이터 영속성**: 애플리케이션을 재시작해도 시료·주문 데이터가 유지되어야 한다.
- **UI**: 콘솔 기반, 메뉴 번호 입력 방식. 정확한 화면 레이아웃은 명세상 자유("화면 구성은 자유롭게 결정하여
  표기")이나, 각 메뉴가 위 기능 요구사항을 모두 노출해야 한다.
- **테스트 가능성**: 상태 전이, 생산량 계산(수율/올림), FIFO 큐 순서는 콘솔 I/O와 분리된 단위 테스트로 검증
  가능해야 한다. (`CLAUDE.md`의 아키텍처 가이드 참고)
- **코드 품질**: Model/Controller/View 계층 분리, Clean Code, 의미 있는 커밋 이력 유지 — 이번 미션([미션2])의
  평가 기준.
